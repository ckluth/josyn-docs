site-builder — JOSYN Documentation Pipeline

A three-part toolchain that assembles the scattered markdown documentation across all JOSYN repositories into a single, navigable HTML site.

The central design idea: where a document lives and where it appears to readers are two separate concerns. Documents stay in their natural home — the repo and folder they belong to. The site-map.yaml defines the logical site structure independently: a human-authored folder tree defines the navigation, and mapping rules assign documents to nodes in that tree. Reorganising the site never touches a source file. Adding a new repo's docs to the site is a one-line change in the YAML.

A second separation runs across the toolchain itself: discovery, semantic enrichment, logical assembly, and HTML rendering are four distinct steps, each with its own cadence and ownership. The indexer knows about repos. The AI enrichment pass knows about meaning. The assembler knows about structure. The renderer knows about HTML. None reaches into the others' concerns.

The pipeline

Step 1 — Index                Step 2 — Assemble              Step 3 — Render
─────────────────────         ──────────────────────         ─────────────────
Walk all JOSYN repos          Read docs-index.json           Take the staged
Find every *.md file          and site-map.yaml              markdown folder
                              Apply skip / section /         Convert to HTML
Produce:                      override rules                 site with nav,
  docs-index.json      ──►    Copy files to staging    ──►   CSS, responsive
  (with git dates,           Generate landing page           design
  preserved semantic          Generate search page
  fields)                                                    josyn-docs/

Run order: indexer → (AI enrichment) → assembler → (inspect staging) → renderer.

The assembler and renderer are deliberately separate steps: after assembling you can inspect the staging folder (temp\josyn-site-staging) before committing to the HTML output. Only run the renderer when the staged content looks right.

Parts

Part Entry point What it does
site-builder-indexer site-builder-indexer.cmd Scans repos → docs-index.json
site-builder-assembler site-builder-assembler.cmd Stages files + calls renderer
site-builder-renderer Called by assembler; also standalone via site-builder-renderer.cmd Markdown folder → HTML site

Configuration files

Both live in josyn-platform/docs/ — they are editorial artefacts, not tools:

File Purpose
docs-index.json Master index of all docs. Produced by the indexer, enriched by AI, consumed by the assembler.
site-map.yaml Assembly rules: sections, skip lists, per-file overrides, reading paths. Hand-maintained.
site-map-authoring.md How to express intent in site-map.yaml — section descriptions, groups, reading paths.

Typical workflow

Full rebuild (after doc changes)

1. site-builder-indexer.cmd      — update docs-index.json
2. (AI enrichment pass if new docs were added — see enrich-docs-index-prompt.md)
3. site-builder-assembler.cmd    — stage files → temp\josyn-site-staging\
4. (inspect staging if needed)
5. site-builder-renderer.cmd     — render staged folder → josyn-docs\
6. git commit in josyn-docs       — publish

First-time setup

1. site-builder-indexer.cmd
2. Run AI enrichment (josyn-platform/docs/enrich-docs-index-prompt.md)
3. Edit josyn-platform/docs/site-map.yaml — design the tree, write mapping rules,
   curate reading paths (see site-map-authoring.md)
4. site-builder-assembler.cmd
5. site-builder-renderer.cmd

Adjust site structure (no doc changes)

1. Edit josyn-platform/docs/site-map.yaml  (tree, mapping, overrides, reading-paths)
   See site-map-authoring.md for guidance.
2. site-builder-assembler.cmd
3. site-builder-renderer.cmd

Dependencies

  • pwsh (PowerShell 7+)
  • dotnet SDK — for running the renderer
  • All JOSYN repos cloned as siblings under the same root directory (e.g. C:\DevGit)
  • cfg/cfg-detect-root.cmd — machine-specific root path resolution

Output

The site is written to josyn-docs/ (defined in site-map.yaml under site.output). It is a self-contained static HTML site — no server required, browsable from file://. It is also published via GitHub Pages.

Azure DevOps pipeline readiness

Not yet done — future task.

The PS1 scripts are already CI-friendly (explicit path parameters, clean exit codes, no interactive elements). The .cmd wrappers and path detection are not. The following changes are needed before this pipeline can run unattended on an Azure DevOps agent.

1. Remove pause from all .cmd wrappers

Every .cmd ends with pause, which hangs a CI agent indefinitely. Replace with a conditional that only pauses on a developer machine:

IF "%TF_BUILD%"=="" pause

Affects: site-builder-indexer.cmd, site-builder-assembler.cmd, site-builder-renderer.cmd.

2. Make cfg-detect-root.cmd CI-aware

cfg-detect-root.cmd resolves the ROOT path by matching COMPUTERNAME against known machine name prefixes. An Azure DevOps agent has an unknown hostname and falls through to the hardcoded developer default (C:\DevGit), which will not exist on the agent.

Fix: detect the TF_BUILD environment variable (set automatically by Azure DevOps) and derive ROOT from a pipeline variable or from $(Build.SourcesDirectory):

IF DEFINED TF_BUILD ( SET "ROOT=$(Build.SourcesDirectory)" & GOTO :root_resolved )

Alternatively, bypass cfg-detect-root.cmd entirely in CI by calling the PS1 scripts directly with explicit -ReposRoot and -StagingDir arguments — which the scripts already support.

3. Multi-repo checkout in the pipeline YAML

Azure DevOps does not automatically check out all JOSYN repositories. The pipeline YAML must declare each repo as a resource and check them all out into a common parent directory so that repos-root resolves correctly.

resources:
  repositories:
    - repository: josyn-platform
      type: git
      name: josyn-platform
    - repository: josyn-foundation
      ...

Nine repos in total (see cfg/cfg-docs-repos-list.cmd for the full list).

4. Commit and push josyn-docs

The pipeline must commit and push the generated HTML output to josyn-docs. This requires a service connection with write access to the repo, and a post-build step that runs:

git -C josyn-docs add .
git -C josyn-docs commit -m "docs: site rebuild [skip ci]"
git -C josyn-docs push

5. .NET SDK version

The renderer (site-builder-renderer) targets .NET 10. Verify the Azure DevOps agent image includes .NET 10 SDK, or add a UseDotNet task to the pipeline YAML.

Preferred CI approach

Rather than calling the .cmd wrappers, a CI pipeline should call the PS1 scripts directly with fully explicit paths. The assembler and renderer are two separate tasks, matching the two-step design:

- task: PowerShell@2
  displayName: Assemble staging folder
  inputs:
    filePath: site-builder/site-builder-assembler/.internal/site-builder-assembler.ps1
    arguments: >
      -SiteMap "$(Build.SourcesDirectory)/josyn-platform/docs/site-map.yaml"
      -ReposRoot "$(Build.SourcesDirectory)"
      -StagingDir "$(Agent.TempDirectory)/josyn-site-staging"
      -NoBuild

- task: CmdLine@2
  displayName: Render staged folder to josyn-docs
  inputs:
    script: >
      dotnet run --project
      site-builder/site-builder-renderer/DocGenerator.csproj --
      "$(Agent.TempDirectory)/josyn-site-staging"
      "$(Build.SourcesDirectory)/josyn-docs"
      "JOSYN Documentation"

This bypasses all .cmd / cfg-detect-root concerns entirely, and makes the assemble-then-render split explicit in the pipeline log.

Design notes

Why a centralised index instead of frontmatter?

Most JOSYN docs are README.md files shown on GitHub repo landing pages. YAML frontmatter is not hidden by GitHub's markdown renderer — it would degrade every repo homepage. The centralised index keeps metadata separate from content, with the indexer as the synchronisation bridge.

Why separate indexer, assembler, and renderer?

Each step has a different run cadence:

  • Index: run when files are added or changed in source repos
  • Enrichment: run (AI-assisted) when new docs have no semantic metadata
  • Assemble+render: run whenever the site needs to be published

The separation lets you rerun only what changed.
