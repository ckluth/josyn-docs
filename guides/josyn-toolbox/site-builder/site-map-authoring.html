site-map.yaml — Authoring Guide

How to tell the assembler what you really want. This guide covers site-map.yaml v2. The schema changed significantly from v1 (sections/include/groups) to v2 (tree/mapping/overrides). If you see those old keys in a file, it is out of date.

The mental model

The site-assembler is a two-layer system:

  • Placement answers: which docs belong in which folder? — driven by mapping and overrides, automated, low maintenance
  • Structure answers: what folders exist and how are they nested? — driven by tree, authored by you
  • Navigation answers: how does a human find their way in? — driven by reading-paths, authored by you

You never list individual documents. You describe structure and intent. Rules handle placement.

The five sections of site-map.yaml

0. site — identity and paths

Controls the title, intro text, and all paths used by the assembler.

site:
  title: "JOSYN Documentation"         # H1 of the landing page; also the HTML <title>
  intro: "One paragraph of context."   # Optional. Shown below the H1 on the landing page. Omit or leave blank to suppress.
  docs-index: "C:/DevGit/josyn-platform/docs/docs-index.json"
  repos-root: "C:/DevGit"
  output:     "C:/DevGit/josyn-docs"
  staging:    "C:/DevGit/temp/josyn-site-staging"
  • title is required.
  • intro is optional. If absent or empty, no intro paragraph is rendered.

1. tree — folder hierarchy

Defines every folder that can receive documents. Indentation controls nesting.

tree:
  - architecture:
      - platform
      - solution-architecture
      - components
      - decisions:
          - platform
          - backend
  - reference
  - repos
  - guides
  - notes
  - unsorted:
      catch-all: true
  • A node with children uses the name: syntax followed by indented child nodes.
  • A leaf node uses the - name syntax.
  • Exactly one node should have catch-all: true. Documents that match no mapping rule land here (with a warning).
  • The path of a node is its full slash-separated ancestry: architecture/decisions/platform.

2. mapping — placement rules

Evaluated top-to-bottom. First matching rule wins.

mapping:
  - { type: agent-instruction, skip: true }
  - { type: decision, repo: josyn-platform, to: architecture/decisions/platform }
  - { type: decision, to: architecture/decisions/platform }
  • to: places matching docs at that tree path. The value must exactly match a path in tree.
  • skip: true excludes matching docs from the site entirely.
  • Fields within a rule are AND. Multiple rule objects are OR (first match wins).
  • Supported match fields: type, repo, state, folder-context. Values support * wildcard.

3. overrides — per-file exceptions

Applied before mapping rules. Useful for one-off placements or exclusions.

overrides:
  - { repo: josyn-platform, path: README.md, to: repos }
  - { repo: josyn-platform, path: MAINTAINERS.md, skip: true }
  • Match is exact: both repo and path must match.
  • Same to: / skip: semantics as mapping rules.

4. reading-paths — landing page navigation cards

Rendered as "Who are you?" entry cards on the landing page. Each path is a persona with a short curated link list.

If reading-paths is empty (reading-paths: []), the entire "Who are you?" section is omitted from the landing page.

reading-paths:
  - perspective: maintainer-architect
    label: "Maintainer — Architect"
    intro: "You design the platform and govern its decisions."
    links:
      - { repo: josyn-platform, path: architecture/overview.md }
      - { section: architecture/decisions }
      - { repo: josyn-platform, path: ROADMAP.md }

Link types:

  • { repo: x, path: y } — links to a specific indexed document.
  • { section: path } — links to a section's index page. path is the full tree path, e.g. architecture/decisions, not just decisions.

Critical rule: whenever you move a section in tree, search reading-paths for any { section: ... } reference to that section and update it. reading-paths holds hard path references — they do not update automatically.

Keep each reading path to three to five links. It is a starting point, not an index.

When to use what

Question Tool
I want a new folder in the site Add a node to tree
I want docs of type X to land in folder Y Add a rule to mapping
One specific doc belongs somewhere unusual Add an entry to overrides
A doc should never appear on the site skip: true in mapping or overrides
A new persona needs a starting point on the landing page Add an entry to reading-paths
I moved a section in the tree Update tree, mapping rules that reference it, and any { section: ... } links in reading-paths

What you do NOT need to do

  • You do not list individual documents. Rules place them automatically.
  • You do not re-run the indexer after editing site-map.yaml. Only the assembler reads it.

Workflow

  1. Edit docs/site-map.yaml
  2. If you moved or renamed a section — grep for its old path in reading-paths and fix references
  3. Run site-builder-assembler.cmd
  4. Inspect the staging folder and the generated _index.md files
  5. Adjust until the structure and landing page communicate what you intended

Reference — annotated complete example

tree:
  - architecture:
      - platform          # top-level docs about the platform architecture
      - decisions:
          - platform      # ADRs from josyn-platform
          - backend       # ADRs from josyn-backend
  - reference
  - unsorted:
      catch-all: true     # unmatched docs land here with a warning

mapping:
  - { type: agent-instruction, skip: true }         # never show AGENTS.md on the site
  - { state: superseded, skip: true }               # hide retired docs
  - { type: decision, repo: josyn-platform, to: architecture/decisions/platform }
  - { type: decision, repo: josyn-backend,  to: architecture/decisions/backend  }
  - { type: decision, to: architecture/decisions/platform }  # fallback for other repos
  - { type: reference, to: reference }

overrides:
  - { repo: josyn-platform, path: MAINTAINERS.md, skip: true }

reading-paths:
  - perspective: maintainer-architect
    label: "Maintainer — Architect"
    intro: "You design the platform and govern its decisions."
    links:
      - { repo: josyn-platform, path: architecture/overview.md }
      - { section: architecture/decisions }   # full tree path, not just "decisions"
