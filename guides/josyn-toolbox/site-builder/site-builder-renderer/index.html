site-builder-renderer

A C# tool that converts a folder of markdown files into a self-contained, static HTML site with a collapsible sidebar navigation, responsive design, a shared CSS stylesheet, and client-side Mermaid diagram rendering.

Part of the site-builder toolchain. Normally called by site-builder-assembler — run it standalone only for single-folder use cases.

What it does

  1. Discovers all *.md files in the source directory (recursive).
  2. Builds a collapsible navigation tree from the folder structure (see below).
  3. For each page:
    • Optionally prepends a small origin line (the source file path within staging).
    • Parses markdown using Markdig (with advanced extensions: tables, code, footnotes, etc.)
    • Renders fenced code blocks tagged mermaid as diagram containers; all other code blocks render normally.
    • Rewrites .md links to .html links (preserving relative paths).
    • Wraps content in the HTML template with the generated nav sidebar.
  4. Writes one .html file per .md file; README.md becomes index.html.
  5. Writes a single site.css to the output root.
  6. Copies mermaid.min.js from the vendored vendor/ directory to the output root.

Mermaid diagrams

Authoring

Any fenced code block tagged mermaid is rendered as a Mermaid diagram:

```mermaid
flowchart LR
    Scheduler --> JobProcess
    JobProcess --> Scheduler
```

No special front-matter or configuration required. All Mermaid diagram types supported by the vendored version are available (flowcharts, sequence diagrams, class diagrams, etc.).

How it works

Four parts cooperate to deliver diagram rendering:

  1. MermaidCodeBlockRenderer — registered into the Markdig pipeline at render time by swapping out Markdig's default CodeBlockRenderer. Fenced blocks tagged mermaid are emitted as <div class="mermaid"> containing the escaped diagram source. All other code blocks are passed unchanged to the original renderer.

  2. HtmlTemplate.MermaidScript — injects two script tags at the bottom of every page:

    • <script src="...mermaid.min.js"> — loads the vendored library (path is adjusted per-page depth so it always resolves to the output root copy).
    • An inline <script> that calls mermaid.initialize({startOnLoad:false, theme:'dark'}) followed by mermaid.run({querySelector:'.mermaid'}). The explicit run() call avoids the race condition that startOnLoad:true can trigger.

  3. SiteBuilder.Build — after writing all HTML pages and site.css, copies vendor/mermaid.min.js (from AppContext.BaseDirectory/vendor/) to the output root as mermaid.min.js. If the file is absent, the copy is skipped silently.

  4. PageRenderer — computes the correct relative prefix for each page (based on its depth in the output tree) so that both site.css and mermaid.min.js resolve correctly regardless of how many directory levels deep a page sits.

Why vendored

The Mermaid library is checked in to vendor/mermaid.min.js and declared in DocGenerator.csproj as a <Content> item with CopyToOutputDirectory: PreserveNewest. This approach was chosen deliberately:

  • No CDN — the generated site works fully offline and from file:// without any network dependency.
  • No npm / Node.js — the renderer has one runtime dependency (Markdig). Introducing a JS build step for a single library would add disproportionate tooling overhead.
  • Pinned version — the library in the repo is the library in the site. No surprise upgrades, no version drift between developer machines.

Current vendored version: Mermaid 11.15.0.

Risks and worst-case scenarios

Risk Worst case Mitigation
No automatic updates — the vendored file is pinned and never updated by tooling. A security vulnerability or rendering bug in Mermaid goes unnoticed until manually discovered. Periodically check the Mermaid changelog. Replace vendor/mermaid.min.js and update the version note above when upgrading.
Silent absence — if vendor/mermaid.min.js is missing from the build output (e.g., <Content> removed from csproj, or a stripped publish profile), the copy in SiteBuilder.Build is skipped silently. All pages load, the <script src="mermaid.min.js"> tag is present, but the file returns 404. Diagrams show raw escaped text inside an unstyled <div>. The build will still succeed with no error. Verify by checking that mermaid.min.js exists in the output root after rendering.
No fallback for invalid syntax — invalid Mermaid source is already past the Markdig stage when it reaches the browser; there is no <pre><code> fallback. Mermaid renders an error <div> in place of the diagram. The rest of the page is unaffected. No action needed at the tooling level; authors must fix the diagram source.
Dark theme hardcoded — mermaid.initialize({theme:'dark'}) is embedded in HtmlTemplate.MermaidScript. A site rendered with a light CSS theme will have dark Mermaid diagrams that clash visually. Change the theme value in HtmlTemplate.MermaidScript. This is a code change, not a runtime option.
HTML-escaped source in the div — MermaidCodeBlockRenderer uses escape: true when writing raw lines, so characters like <, >, and & are HTML-encoded in the <div> source. Mermaid reads diagram source from .textContent, which automatically decodes HTML entities — so rendering is correct. If you ever debug the raw HTML and see encoded entities, this is expected and not a bug. No action needed; the escaping is intentional and correct.

Usage

Via assembler (normal)

The assembler calls the renderer automatically. No manual invocation needed.

Standalone

site-builder-renderer.cmd

Edit the SOURCE, OUTPUT, and TITLE variables at the top of the .cmd file.

Or call directly:

dotnet run --project DocGenerator.csproj -- <source-dir> <output-dir> [<site-title>] [--no-show-origin]

Arguments

Position / Flag Name Default Description
1 source-dir (required) Directory containing .md files to render
2 output-dir (required) Directory where HTML files are written (created if absent)
3 site-title folder name Title shown in the nav header and <title> tags
--no-show-origin off Suppress the origin line at the top of each page

Navigation structure

The nav sidebar is built from the folder structure of the source directory:

  • A root README.md → index.html becomes the Home link (plain, not collapsible).
  • Each subfolder that contains a README.md → index.html becomes a nav node.
  • Nav nodes are collapsible using native <details>/<summary> — no JavaScript required.
  • The node containing the current page starts expanded (open); all others start collapsed.
  • Depth is limited to 2 levels below the root. Folders deeper than that are not shown as nav nodes but remain accessible via the section index pages.

Example nav structure for a site with architecture and repos sections:

JOSYN Documentation          ← home link
▶ Architecture               ← collapsible section (depth 1)
  Overview
  ▶ Platform                 ← collapsible sub-section (depth 2)
    Overview
  ▶ Solution Architecture
    Overview
▶ Repos
  Overview
  ▶ Josyn Foundation
    Overview
  ▶ Josyn Backend
    Overview

Origin line

When enabled (default), a small monospace line appears at the top of each page showing the source file's path within the staging directory:

architecture/platform/josyn-platform/architecture/overview.md

Useful during review to trace a rendered page back to its source. Disable with --no-show-origin.

Link rewriting

All relative .md links in the source markdown are rewritten to .html links in the output. Directory-style links (ending with /) are resolved to the index.html in that directory. External and anchor-only links are left unchanged.

Source files

File Responsibility
Program.cs Argument validation, entry point
SiteBuilder.cs File discovery, recursive nav tree construction, orchestration, vendor file copy
PageRenderer.cs Markdig parsing, link rewriting, nav HTML rendering, HTML assembly
HtmlTemplate.cs HTML template and CSS (all visual concerns), Mermaid script injection
MermaidCodeBlockRenderer.cs Markdig renderer override — routes mermaid fenced blocks to <div class="mermaid">
vendor/mermaid.min.js Vendored Mermaid library (11.15.0) — copied to output root at render time

Dependencies

  • .NET 10 SDK
  • NuGet: Markdig (markdown parsing with advanced extensions)
  • vendor/mermaid.min.js (vendored; checked in — no network or npm required)
