ADR-010 — Environment Separation and Installation Model

Date: 2026-06-05 Status: Proposal — not yet fully designed; enough context to continue later

Context

JOSYN operates in three environments: PROD, INT, DEV.

During bootstrapping design (ADR-009 follow-up) a key insight emerged:

A server is already within an environment by nature of its installation.

This means environment selection is an installation-time concern, not a runtime concern. There is no env-switch flag, no [DEV] / [PROD] section in a shared config file. Each installation has its own josyn.bootstrap.ini containing the correct values for exactly one environment.

Central decision (to be confirmed)

Environment = Installation. One installation, one environment, one bootstrap file.

Consequences:

  • josyn.bootstrap.ini is flat (no sections). The INI sections capability of IniDictionarySerializer is not used for environment switching.
  • No runtime environment selection logic in josyn-backend.
  • Deploying to a new environment means creating a new installation with its own bootstrap file.
  • The risk of PROD/DEV cross-contamination is eliminated structurally, not by runtime guards.

Open questions (to resolve when this ADR is formally written)

  1. What is the installation unit? A directory? A Windows service instance? A container? The answer shapes what "one installation" means in practice.

  2. How does DEV work on a developer machine? A developer may run DEV and INT locally. Are these two separate installation folders? Two separate service instances? Is there a convention for local developer setups?

  3. Are there any truly environment-invariant bootstrap values? E.g., log format, culture, timeout values — things that would be identical across all environments. If yes: do they live in the same bootstrap file, or separately?

  4. How are bootstrap files managed and deployed? Checked into a secrets store? Generated by a deployment pipeline? Managed manually? This is an operational question but shapes what the file format must support. ⚠️ Installation note: The initial provisioning of josyn.bootstrap.ini must be explicitly addressed in the installation process — it is a required file that must exist before the backend can start, is never committed to source control, and must be created as part of every new environment setup.

  5. Interaction with ADR-009 adapters: The HAEVG adapter (context provider) may itself be environment-aware — e.g., it may call a different HAEVG config endpoint for PROD vs. INT. Is that the adapter's own concern, or does JOSYN pass the active environment as a parameter to the adapter?

Relation to other ADRs

  • ADR-009 (Runtime Context Provider Pattern): assumes a single flat bootstrap file per installation. This ADR confirms that assumption and explains why.
  • ADR-010 (this): defines the installation model that makes ADR-009's assumption valid.
