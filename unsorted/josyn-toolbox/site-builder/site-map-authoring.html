site-map.yaml — Authoring Guide

How to tell the assembler what you really want.

The mental model

The site-assembler is a two-layer system:

  • Rules answer: which docs belong in which section? — automated, low maintenance
  • Intent answers: what does this section mean, and how should a human navigate it? — authored by you

You never need to list individual documents. You only need to describe structure and meaning. The rules handle placement. You handle purpose.

The three tools you have

1. Section description

"What is this section and who reads it?"

Add a description field to any section in site-map.yaml. One or two sentences. Rendered as the intro paragraph on the section's overview page.

- title: Repos
  folder: repos
  description: "One entry per repository — what it is, what it contains, and how it relates to the rest of the platform."
  include: ...

Use this whenever a human landing on the section page would otherwise have no context.

2. Groups within a section

"These docs belong together and deserve a shared heading."

Add a groups list to a section. Each group has a label and a match rule — the same rule syntax used in include. The assembler assigns docs to groups and renders them under sub-headings.

  groups:
    - label: "Core repos"
      match: { repo: josyn-foundation }
    - label: "Protocol layer"
      match: { repo: josyn-jap }
    - label: "Platform"
      match: { repo: josyn-platform }
      description: "Cross-cutting architecture and standards documents."

You do not need to cover every doc. Docs matching no group collect in an implicit Other block at the bottom.

Use groups when a section contains docs from distinct areas that would be confusing as one flat list.

3. Reading paths

"If you are X, start here."

Defined under reading-paths in site-map.yaml. Each path has a label, an intro sentence, and a short list of links (doc references or section links). Rendered as entry cards on the landing page.

- perspective: job-developer
  label: "Job Developer"
  intro: "You write jobs that run on the JOSYN platform."
  links:
    - { section: guides }
    - { section: repos }

Keep each reading path short — three to five links. It is a starting point, not an index.

When to use what

Question you are asking Tool
A reader lands on a section and doesn't know what it is description on the section
A section mixes docs from different repos or topics groups on the section
A group of docs within a section needs its own context sentence description on a group
A new person doesn't know where to start in the whole site reading-paths
A specific doc belongs in a different section than the rules place it overrides
A doc should not appear on the site at all skip

What you do NOT need to do

  • You do not list individual documents. Rules place them automatically.
  • You do not assign docs to groups manually. Groups use the same match-rule syntax.
  • You do not re-run the indexer after editing site-map.yaml. Only the assembler reads it.

The workflow

  1. Open josyn-platform/docs/site-map.yaml
  2. For each section — ask: "Would a stranger understand what this section is?" If not, add a description.
  3. For each section — ask: "Is this a confusing mix of unrelated things?" If yes, add groups.
  4. Run site-builder-assembler.cmd — inspect the staged {section}/README.md files
  5. Adjust until the overview pages communicate what you intended

Reference — full section with all intent fields

- title: Repos
  folder: repos
  description: "Per-repository reference summaries and sub-package overviews."
  groups:
    - label: "Foundation layer"
      match: { repo: josyn-foundation }
      description: "Infrastructure primitives — Result pattern, serialisation, IPC transport."
    - label: "Protocol layer"
      match: { repo: josyn-jap }
    - label: "Execution layer"
      match: { repo: josyn-job-host }
    - label: "Scheduler"
      match: { repo: josyn-backend }
    - label: "Platform"
      match: { repo: josyn-platform }
  include:
    - { type: repo-summary }
    - { type: overview, folder-context: "" }
    - { type: overview, folder-context: "josyn-foundation*" }
    - { type: overview, folder-context: "josyn-jap*" }

All intent fields are optional. A section without description or groups behaves exactly as today — a flat table, no intro.
